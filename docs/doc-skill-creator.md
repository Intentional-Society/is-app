# skill-creator (vendored upstream skill)

Anthropic's [skill-creator](https://github.com/anthropics/skills/tree/main/skills/skill-creator)
skill — the workflow the team uses to build, eval, and description-optimize Claude Code Skills
(it built `/commit`, `/pr`, `/ship`; see `docs/spec-portable-ai-procedures.md`) — is vendored at
`.claude/skills/skill-creator/` so every teammate has it on clone with zero setup.

Design, alternatives considered, and the full PR sequence: `docs/plan-skill-creator-vendoring.md`.

## How the vendoring works

- The copy is **verbatim upstream** (including `LICENSE.txt`, Apache-2.0), pinned to the last
  upstream commit that touched the subdirectory. The pin lives in
  `.claude/skills/skill-creator/UPSTREAM.md` — the only non-upstream file in that directory,
  regenerated by the update script. Never hand-edit the vendored files.
- The skill is model-invokable (no `disable-model-invocation: true`): it is a non-side-effecting
  authoring tool, so natural-language invocation ("help me create a skill") is deliberate.

## Updating

```
node scripts/update-skill-creator.mjs --check   # compare pin vs upstream; exit 0 current, 2 behind
node scripts/update-skill-creator.mjs           # sync to latest upstream, rewrite UPSTREAM.md
node scripts/update-skill-creator.mjs --sha=<sha>  # sync to a specific upstream commit
```

Requires `gh` (authenticated) and `tar` (ships with Windows 10+/macOS/Linux). The script never
commits. After a refresh: review the diff, run the acceptance evals in
`evals/skill-creator.evals.json` and capture results in the commit's Test Plan, then ship via
`/commit`.

You don't have to remember to run `--check`: a monthly workflow
(`.github/workflows/skill-creator-drift.yml`, 05:07 UTC on the 1st; manual via
`gh workflow run skill-creator-drift.yml`) runs it and opens a tracking issue — labeled
`dependencies` + `skill-creator-drift`, one open at a time (deduped by the qualifier label) —
when the pin falls behind upstream. It never syncs files or opens PRs; refreshing stays the
human-run procedure above.

## Running its scripts (authoring time only)

The bundled `scripts/*.py` and `eval-viewer/` need **Python 3 with PyYAML**
(`python -m pip install --user pyyaml`). Nothing in CI or the app runs them — Python is a
human-at-the-keyboard authoring prerequisite only.

Known divergence, expected: upstream's `scripts/quick_validate.py` enforces a strict frontmatter
key allowlist and rejects the Claude Code key `disable-model-invocation` used by team skills.
That validator is an upstream authoring aid, not this repo's gate; the repo's own structural
checks are (per the plan, PR 2) a Vitest test asserting the team's spec.

Generated run artifacts (`__pycache__/`, packaged `*.skill` files) are gitignored — never commit
them.
